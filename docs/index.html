Zero-Knowledge Identity Chain - Decentralized Identity Ledger

Overview

Veritas-Chain provides a decentralized identity management system with zero-knowledge proof verification. This API allows you to register identities, issue permission tokens, and perform selective field verification without revealing the actual data.

Base URL
http://localhost:8081/api

Keypair Generation and Signing

Before using the API endpoints that require signatures, you need to understand how to generate Ed25519 keypairs and create signatures.

Generating Ed25519 Keypairs

Use Node.js crypto module to generate secure Ed25519 keypairs:

import crypto from 'crypto';
import { promisify } from 'util';

const generateKeyPair = promisify(crypto.generateKeyPair);

// Generate a new Ed25519 keypair
const { publicKey, privateKey } = await generateKeyPair('ed25519', {
  publicKeyEncoding: { type: 'spki', format: 'pem' },
  privateKeyEncoding: { type: 'pkcs8', format: 'pem' }
});

console.log('Public Key:\n', publicKey);
console.log('Private Key:\n', privateKey);

Creating Signatures

Signatures are created by signing specific data with a registrar's private key:

import crypto from 'crypto';

// For registration: sign the data object
const data = {
  name: "John Doe",
  dob: "1990-05-15",
  address: "123 Main St"
};
const dataString = JSON.stringify(data);

// Sign with registrar's private key
const signature = crypto.sign(null, Buffer.from(dataString), registrarPrivateKey);
const signatureBase64 = signature.toString('base64');

console.log('Signature:', signatureBase64);

Verifying Signatures

Signatures can be verified using the public key:

import crypto from 'crypto';

const isValid = crypto.verify(
  null,
  Buffer.from(dataString),
  registrarPublicKey,
  Buffer.from(signatureBase64, 'base64')
);

console.log('Signature valid:', isValid);
Important: Signatures Must Be From Other Registrars
When submitting requests, the registrarPrivateKey field is used for authentication of the submitter. The signatures array must contain signatures from OTHER authorized registrars (not the submitter). Each signature signs the specific data being submitted (e.g., the data object for registration, newData for updates, or public key for key registration).

POST /register

Register a new identity on the blockchain (newborn registration)

Request Body

Parameter Type Required Description
data object Yes Identity data fields (name, dob, address, bloodGroup, etc.)
registrarPrivateKey string Yes Private key of the submitting registrar (used for authentication, not counted as a signature)
signatures array Yes Array of base64-encoded signatures from OTHER authorized registrars. Each signature must sign JSON.stringify(data). Must meet config requirement (default: 2)
parentKeys array Yes Array of parent private keys (at least 1 required, both preferred)

Request

Linux/Mac
Windows
curl -X POST http://localhost:8081/api/register \
  -H "Content-Type: application/json" \
  -d '{
    "data": {
      "name": "John Doe",
      "dob": "1990-05-15",
      "address": "123 Main St",
      "bloodGroup": "O+",
      "fatherName": "Father Doe",
      "motherName": "Mother Doe"
    },
    "registrarPrivateKey": "-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----",
    "signatures": [
      "base64-encoded-signature-from-registrar2",
      "base64-encoded-signature-from-registrar3"
    ],
    "parentKeys": [
      "-----BEGIN PRIVATE KEY-----\nfather-private-key\n-----END PRIVATE KEY-----",
      "-----BEGIN PRIVATE KEY-----\nmother-private-key\n-----END PRIVATE KEY-----"
    ]
  }'
curl.exe -X POST http://localhost:8081/api/register ^
  -H "Content-Type: application/json" ^
  -d "{\"data\": {\"name\": \"John Doe\", \"dob\": \"1990-05-15\", \"address\": \"123 Main St\", \"bloodGroup\": \"O+\", \"fatherName\": \"Father Doe\", \"motherName\": \"Mother Doe\"}, \"registrarPrivateKey\": \"-----BEGIN PRIVATE KEY-----\\n...\\n-----END PRIVATE KEY-----\", \"signatures\": [\"base64-sig-1\", \"base64-sig-2\"], \"parentKeys\": [\"-----BEGIN PRIVATE KEY-----\\nfather-key\\n-----END PRIVATE KEY-----\", \"-----BEGIN PRIVATE KEY-----\\nmother-key\\n-----END PRIVATE KEY-----\"]}"

Response

  "success": true,
  "blockHash": "...",
  "ownerPublicKey": "...",
  "ownerPrivateKey": "...",
  "encryptionKey": "..."
}
Signatures from Other Registrars
The submitting registrar provides their private key for authentication. The signatures array must contain signatures from OTHER authorized registrars (not the submitter). Each signature must sign the exact data object being registered: JSON.stringify(data).

Parameter Type Required Description

Request

Linux/Mac
Windows

    "signatures": [
      "base64-encoded-signature-from-existing-registrar1",
      "base64-encoded-signature-from-existing-registrar2",

curl.exe -X POST http://localhost:8081/api/keyregister ^

  -d "{\"newRegistrarPrivateKey\": \"-----BEGIN PRIVATE KEY-----\\n...\\n-----END PRIVATE KEY-----\", \"signatures\": [\"base64-sig-1\", \"base64-sig-2\", \"base64-sig-3\"]}"

Response

{
  "success": true,
  "registrarPublicKey": "-----BEGIN PUBLIC KEY-----\n...\n-----END PUBLIC KEY-----",

Manual Configuration Step Required
After successful approval, you must manually add the returned registrarPublicKey to the KeyRegistry array in config.json to complete the registration. Existing registrars sign the new registrar's public key (computed from the private key) to approve them.

POST /verify

Perform zero-knowledge verification on multiple fields

Request Body

Parameter Type Required Description
blockHash string Yes Hash of the identity block
tokenId string Yes Token ID with permission to access the fields
conditions array Yes Array of condition objects [{field, condition}, ...]
encryptionKey string Yes Base64 encoded encryption key

Request

Linux/Mac
Windows
curl -X POST http://localhost:8081/api/verify \
  -H "Content-Type: application/json" \
  -d '{
    "blockHash": "...",
    "tokenId": "...",
    "conditions": [
      {"field": "dob", "condition": "<= 2007-10-20"},
      {"field": "bloodGroup", "condition": "== O+"}
    ],
    "encryptionKey": "..."
  }'
curl.exe -X POST http://localhost:8081/api/verify ^
  -H "Content-Type: application/json" ^
  -d "{\"blockHash\": \"...\", \"tokenId\": \"...\", \"conditions\": [{\"field\": \"dob\", \"condition\": \"<= 2007-10-20\"}, {\"field\": \"bloodGroup\", \"condition\": \"== O+\"}], \"encryptionKey\": \"...\"}"

Response

{
  "success": true,
  "results": [
    {"field": "dob", "condition": "<= 2007-10-20", "passed": true},
    {"field": "bloodGroup", "condition": "== O+", "passed": true}
  ],
  "allPassed": true
}

Response (if token lacks permission)

{
  "success": false,
  "error": "Token lacks permission for one or more fields",
  "unauthorizedFields": [
    {"field": "dob", "reason": "Token does not have permission for field: dob"}
  ]
}
Zero-Knowledge Proof & Security
The verification validates ALL token permissions BEFORE any decryption occurs. If even one field is unauthorized, the request is rejected with HTTP 403 and no data is decrypted. When authorized, it returns only true/false for each condition without revealing actual field values, preserving privacy while allowing selective disclosure of multiple conditions simultaneously.

Automatic Stage Calculation
The lifecycle stage is automatically determined based on rotation count:
  • Rotation 1: genesis ‚Üí guardian (~age 5)
  • Rotation 2: guardian ‚Üí self (~age 18)
  • Rotations 3-5: self ‚Üí self (additional transitions)
Maximum 5 rotations allowed per identity.

Lifecycle Stages

Stage Keypair Trigger Description
Genesis Genesis Key (Hospital) Newborn registration Initial block creation
Guardian Guardian Key At age 5 Block re-encrypted with guardian's key
Self Personal Key At age 18 Block re-encrypted with personal key
Expired None Death + 3 years grace Block pruned from chain

Security & Privacy

Cryptographic Algorithms

  • Hashing: SHA-512 for block integrity
  • Symmetric Encryption: AES-256-GCM for field-level data encryption
  • Asymmetric Keys: Ed25519 for signatures and authentication
  • Token IDs: Nanoid for secure, URL-safe identifiers

Zero-Knowledge Proofs

The verification endpoint allows checking conditions on encrypted data without revealing the actual values. Only boolean results are returned, preserving privacy while enabling selective disclosure.
